# Sprint 1 Week 3 — Technical Decisions

## 1. Grafana URL Rewrite Strategy

**Decision:** Single `rewriteGrafanaUrl()` utility rewrites Docker-internal `http://grafana:3000` to `VITE_GRAFANA_URL`.

**Why:** Backend returns Docker-internal hostnames (grafana:3000) because it runs inside the Docker network. Browsers can't resolve these. Rather than making the backend URL-aware of the frontend's context, we rewrite at the edge — one function, one env var.

**Production:** Just set `VITE_GRAFANA_URL` to the production Grafana URL at build time.

## 2. Docker node_modules Volume

**Decision:** Named volume `frontend_node_modules:/app/node_modules` prevents host bind mount from clobbering container deps.

**Why:** Without this, `./frontend:/app` would overlay the container's `/app/node_modules` (built for Linux/Alpine) with the host's `node_modules` (built for macOS). Native modules (e.g., esbuild) would fail. The named volume keeps container deps isolated.

## 3. Iframe Refresh Strategy

**Decision:** Grafana handles auto-refresh via `&refresh=5s` query param appended by `GrafanaEmbed`. Manual refresh uses React key toggling.

**Why:** No polling or WebSocket complexity needed. Grafana's built-in refresh is reliable and configurable per-panel. The key toggle trick (`key={refreshKey}`) is idiomatic React — changing the key unmounts and remounts the iframe, forcing a fresh load.

## 4. Tailwind via Vite Plugin

**Decision:** Using `@tailwindcss/vite` plugin instead of PostCSS config.

**Why:** Tailwind v4 provides a first-party Vite plugin that's faster and simpler than the PostCSS approach. No `tailwind.config.js` or `postcss.config.js` files needed. Just `@import "tailwindcss"` in CSS and add the plugin to `vite.config.ts`.

## 5. TanStack Router (File-based)

**Decision:** File-based routing with `@tanstack/react-router` instead of React Router.

**Why:** Type-safe routes, automatic code splitting, and file-based routing convention. The route tree is auto-generated by the Vite plugin — no manual route registration.

## 6. Zustand Auth Store Placeholder

**Decision:** Empty Zustand store (`authStore.ts`) committed as a placeholder.

**Why:** Week 4 adds authentication. Having the store structure in place means the auth integration will be additive, not a refactor. Zero runtime cost — the store is never rendered.

## 7. React Query for Data Fetching

**Decision:** `@tanstack/react-query` wraps all API calls via thin hook wrappers.

**Why:** Automatic caching, deduplication, background refetching, loading/error states. The hooks (`useDevices`, `useDevice`, `useDeviceEmbedUrls`) are minimal — just query key + fetch function.
